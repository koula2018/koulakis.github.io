Monad transformers and state

Some interesting tools of functional programming are functors, applicative functors and monads. They provide an elegant way to lift your arguments about types to wrapped types such as collections, optional types and IO, without having to explicitly argue about all possible outcomes. Though those tools lead to happy days, one stumbles quickly to the first problem: how do you use the same declarative expressiveness when combining two different monads? Luckily enough, there is already a solution for this problem called monad transformers. Those guys wrap the one monad inside the other and provide a nice way to transfer arguments between them using the lift function.

One nice example of a situation where one would use transformers is that of a state machine which ocassionally receives input and produces output. There one can wrap the IO monad with the state monad and then control both of them with the transformer. At some point I decided that Bender, one of my favorite codingame puzzles, whould be a great case for such a state machine and refactored my F# code to Haskell to give it a try. Event sourcing

Event sourcing is one of those ideas that when you first encounter them you think "This is so simple and awesome! Why haven't I used this stuff before?". It's simplicity lies in the realization that instead of spending time to persist you data in structured tables you can just record each change with an event which contains all relevant information. Afterwards, when you want to read the value of some entity, you can go through the relevant events and decide what its last status was. This pattern is a close friend to the command query responsibility segregation (CQRS) and one often meets them implemented together.

One simple example of such a process is git. You save each change of your code in a commit and then you can use your commit history to recover the latest version of your code. Aside from that, you can always go back to a past version of your code and also analyse your workflow history.

Event sourcing has some amazing advantages when compared with other ways of persisting data such as the relational model:

  • the history of all the transactions is saved and can be used to perform any kind of business analysis and data mining; you even keep the information you once considered useless and is now helping you on your marketing strategies
  • the data is immutable, thus you loose the ability to create a chaos by changing or deleting the wrong value
  • if something goes wrong, you can go through the history and discover weaknesses and bugs in your software
  • data saved in event stores

Of course, like almost any choice in programming, it comes along with several disadvantages and one has to decide wizely whether this is the right choice for his type of data:

  • it costs time to build the extra infrastructure that manages the persistence and projections of events
  • there is no uniform way, like sql, to query your data
  • when the amount of events becomes large one has to begin taking snapshots to improve performance

Set Theory

Set theory has always been fascinating to me because it is the most natural example of a mathematical theory whose main questions are not provable using the traditional foundations of mathematics. A large portion of those questions are attempting to clarify the nature of the real numbers, for example trying to determine which sets of reals we can assign probabilities to. Interestingly, most of those questions can be categorized by using a sequence of different levels of infinity called large cardinals. This has been the topic of my master thesis, supervised by Athanassios Tzouvaras, which is actually a short survey presenting the core properties of several "popular" large cardinals. Later on, I got to work with Ralf Schindler, my PhD suprvisor, on a problem relating a category of large cardinals with a set of properties of the real numbers.
